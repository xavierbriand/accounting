## Change

- **Sonnet implementer killed mid-slice-4 by the harness; recovery added unstructured Opus inline work.** The Sonnet agent completed slices 2 + 3 (committed) and was mid-slice-4 (8 files modified, all per spec; tests passing) when the harness terminated it. Opus recovered by inspecting `git status`, running the test suite (250 passing, 3 acceptance scenarios still red as expected for that point in the TDD cycle), and committing slice 4 as Sonnet would have. Slices 5–8 were then implemented by Opus directly. **This worked, but skipped the structured Sonnet return report** (Section 8 of the PR template was filled in by Opus instead). Two consequences: (a) deviations were recorded by Opus retroactively rather than by the implementer in real time; (b) the Sonnet vs Opus tier separation is muddied for this story. **Next time:** when Sonnet is killed mid-implementation, prefer to re-spawn Sonnet with a "resume from commit X, complete slices Y..Z" brief over inline Opus continuation. Falls back cleanly to inline work only when re-spawning fails. Action item A.
- **Plan-time review didn't authorise the implicit pixelmatch dev-dependency.** quickpickle 1.11.1 imports `pixelmatch` unconditionally from its ESM bundle but doesn't declare it as a dependency. Sonnet's slice 2 worked around it with a local `node_modules/pixelmatch/` stub (not committed); slice 3 properly added `pixelmatch ^7.1.0` to `devDependencies`. The plan said "no new deps"; the dep was needed but not pre-authorised. P3 retro flagged it as MAJOR. **Next time:** when wiring a new test framework (or any tool with optional/peer/undeclared deps), the Plan-agent stress-test should explicitly probe "what does this tool actually need at runtime" by checking the tool's bundle and listing all imports, not just the declared `dependencies`. For test deps the bar is lower than runtime, but the *authorisation* should still be in the plan. Action item B.
- **Tried `setupFiles: ['tests/features/steps/*.steps.ts']` glob and reverted — vitest doesn't expand globs in setupFiles.** P3 retro #3 suggested the glob; I tried it; suite immediately broke with `Cannot find module '...steps/*.steps.ts'`. Reverted to the literal path. **Next time:** when adopting a P3 retro-suggested config refactor that's not adjacent to a covered behaviour, run a quick smoke test before committing — vitest's globbing semantics in `setupFiles` (no, only in `include`) is the kind of thing CHANGELOG-archaeology won't surface but a test run will. Filed as #58 for proper investigation in Story 3.2 planning. Action item C.
- **Two test-arbitrary fixes folded into slice 6's `feat:` commit instead of a separate `fix:` slot.** fast-check 4 requires 32-bit-float bounds (`fc.float({ min: 0.01, max: 0.99 })` errors out); `fc.date()` occasionally produces an Invalid Date. Both surfaced only when the slice-5 test file actually loaded (slice 5 failed at module import). Folded into slice 6 because they're prerequisites for the green-state, with documentation in the commit body. The TDD-by-intent invariant held (the property tests still measure what they claim to measure), but the rhythm "test fails → feat green" got slightly muddied. Acceptable here because fast-check API discovery is a test-infrastructure concern, not a production-code one — but if a future story discovers a similar issue, consider a small `fix(test):` slot between `test:` and `feat:`. Informal lesson; if it recurs, codify.

- **AskUserQuestion the load-bearing design decisions before Plan-agent.** Story 3.1 used three plan-mode AskUserQuestion items (epic breakdown, validity-window storage, category persistence) before the Plan-agent ran. Reviewers cited the locked decisions when checking design coherence. Recommend adopting as a default for Epic 3's remaining stories: identify the 2–4 decisions that the rest of the plan rests on, lock them with the user, then run Plan-agent on the implementation details.
