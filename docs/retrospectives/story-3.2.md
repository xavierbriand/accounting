# Story 3.2 retrospective

**PR:** [#76](https://github.com/xavierbriand/accounting/pull/76)  **Closed:** pending merge

Second story of Epic 3. Builds the read side of the buffer subsystem (`BufferStateService` + `BufferLedgerQuery` port + `SqliteBufferLedgerQuery` adapter) and locks the bucket→ledger-account convention via an explicit `account` field on `BufferBucket`. Pure Core service mirrors `SplitRulesService` (Story 3.1) — same shape, same purity invariants, same ISO-date keying. First story to ship a real Money-returning Infra port (architecture data-boundary kept clean by handling cents→Money translation in the adapter, not the Core type). 12 commits on the branch (plan + plan-revision + 9 implementation slices + 1 phase-4 refactor + this retro). Test count: 308 → 345 (+37). First story of Epic 3 to use the `plan-reviewer` and `code-reviewer` sub-agents for both phase 2 and phase 4 — both fired cleanly without harness friction.

## Keep

- **AskUserQuestion the load-bearing decisions before any Plan-agent call.** Adopted as a "Try" from the Story 3.1 retro, now confirmed: Phase 1 opened with two AskUserQuestion rounds locking (a) bucket→account mapping (explicit field), (b) API surface (single `getStateAsOf`), (c) state-record shape (balance + target + cap + status), (d) currency policy (single-currency fail-fast); then a second round on (e) asOf-filter semantics, (f) read-port location, (g) duplicate-account rejection. The Plan-agent never re-debated any of these — every reviewer cited the locked answers when checking coherence. Same-day asOf substr bug was caught by the Plan-agent precisely because it had a fixed contract to test against. Pattern continues for Stories 3.3–3.5.
- **Plan-reviewer sub-agent caught the data-boundary violation (`LedgerSums` in Core) and the YAGNI `Money.compare`.** The plan's first draft put `LedgerSums { debitCents, creditCents, currency }` in `src/core/buffers/buffer-state.ts` and added a three-method Money ordering surface (`compare` + `lessThan` + `lessThanOrEqual`). The plan-reviewer flagged both as P3 architecture / engineering-standards violations: the first leaks DB primitives across the Core boundary; the second over-engineers a service that uses only two predicates. Plan rewritten before Sonnet started: port returns `Result<Money>`, adapter owns the cents→Money translation, `Money.compare` dropped. The dedicated reviewer surface (separate sub-agent for plan vs code review, codified in story-maint-13 and story-maint-14) earned its keep on its first epic-3 outing.
- **Phase 2 plan-reviewer also caught the `substr(occurred_at, 1, 10) <= ?` bug.** The original plan used a raw lex-compare against `'YYYY-MM-DD'`. Stored values are `'YYYY-MM-DDTHH:MM:SS+HH:MM'`, so `'2026-04-21T...' > '2026-04-21'` would silently exclude every same-day row. Plan-reviewer recommended `substr(occurred_at, 1, 10) <= ?`. Adopted with explicit "receipt-truth local date" semantics documented in the AC. Caught at Phase 2, not after green tests had already enshrined a wrong predicate.
- **R7 step-wiring policy made explicit at slice-4 plan time.** Plan-reviewer asked: "scenarios 2 and 4 claim to validate SQL-level paths — does the step actually wire the real adapter?" Plan rewrite locked the policy in slice 4: scenarios 1 + 3 use a fake `BufferLedgerQuery`; scenarios 2 + 4 construct the service against `Database(':memory:')` + `runMigrations`. The steps file documents the split in a header comment. Pre-decided, not discovered after green.
- **Property test #1 generator made explicit before Sonnet ran.** Plan-reviewer P2: "if `balance` only generates non-negatives, the negative-balance branch of `'below'` is unexercised." Plan rewrite specified `fc.integer({ min: -10_000_00, max: 10_000_00 })` for the balance generator. The test landed with the negative range. The negative-balance branch in `deriveStatus` is now testpinned, not implicit.
- **Code-reviewer sub-agent caught two property-test honesty issues that would have rotted silently.** Phase 4 retro-check found that the "order-independence" property compared two calls on the same setup (idempotency, not order-independence) and the "monotonicity" property used hard-coded sums on two separate fakes (trivially true by construction; never observed `getStateAsOf` forwarding the date). Both passed the test suite green; both `# fails if` claims were unreachable from the in-process mechanism. Phase 4 refactor commit (`59639e1`) replaced both with the actual service-tier invariants the tests can observe — bucket-order independence and a recording-fake forward-the-date property. SQL-level monotonicity remains covered by the integration tests where the mechanism reaches it. The split between "what the test claims" vs "what the mechanism can validate" is exactly R7's domain; the code-reviewer earns its keep on this kind of subtle drift.
- **`refactor-during-green` allowance held cleanly in slice 8.** Sonnet added two coverage-only tests (29 LOC) for the `ltTargetResult.isFailure` / `lteCapResult.isFailure` defensive guards in `deriveStatus`. No production code touched in slice 8. Within the 20-LOC carve-out spirit, the test additions are local and obvious. R11 doesn't apply (commit isn't empty). Refactor scope was honest.

## Change

Slice 6 added tests for already-green production. The correct label was `test:` (or, better, the tests should have been in `c21bf69` which was the slice-5 paired test commit). Sonnet's deviation note acknowledged the slice-6 ISO_DATE-was-implemented-in-slice-5 reorg but didn't catch that the empty production diff makes `feat:` misleading. Code-reviewer flagged it. Too late to relabel without history rewrite. **Next time:** when re-organising slices during implementation (legitimately, e.g., to avoid a stub), Sonnet should re-derive the commit-type label from the actual diff content, not from the slice number's planned label. Codify in the sonnet-implementer agent spec? Not yet — one occurrence; promote to a rule if it recurs. Action item A.
- **11 commits landed vs R13's 6–10 cap.** Plan target was 8; landed 11. Two over-runs: (a) slices 1+2 were not compressed (plan said "may compress if diff is small"; the diff was small but slicing was kept separate by Sonnet for clarity, which the plan tolerated); (b) the test-only "feat" commit in slice 6 (see above) added a separate commit for already-green code. Acceptable but worth noting: the "may compress" phrasing in the plan deferred the decision to Sonnet; deferred decisions tend to default to no-action. **Next time:** when a slice pair is small enough to compress, the plan should *prescribe* the compression, not permit it. Action item B.
- **Plan AC line and Gherkin/slice spec drifted on the duplicate-account error format.** Plan AC line 37 cited `buffers.<i>.account: duplicate of buffers.<j>.account`. Plan-reviewer flagged that the existing `findDuplicateIndices` helper records only the duplicate's index, not the original's, so the `<j>` part isn't reachable without modifying the helper. Plan rewrite simplified Gherkin scenario 3 and slice 2 to `'duplicate account'` at `path: [i, 'account']` — but the AC prose at line 37 wasn't updated to match. Implementation followed Gherkin/slice (correctly). Code-reviewer caught the doc-only drift in Phase 4. No runtime impact; only a future-reader-confusion risk. **Next time:** when adopting a plan-reviewer suggestion that simplifies a constraint, walk *all* sections of the plan that reference the constraint, not just the slice description. Action item C.

## Try

- **Pre-prescribe slice compression where the plan judges it possible.** The plan's "Slices 1+2 may compress to one commit if the diff is small" left the decision to Sonnet; it didn't compress. For Story 3.3, when the plan judges two adjacent slices to be compressible, the prescription should read "Slices X+Y are bundled in one commit subject `<verb>: A and B`" — no implementer judgement call. Reduces commit count toward the R13 mid-band naturally. If the diff turns out large during implementation, Sonnet can split — but defaulting to compressed shifts the failure mode in the right direction. Drop into the next story's plan as an experiment.
- **Add an "Updated AC?" cross-reference checklist when adopting plan-reviewer rewrites.** When the plan-reviewer says "simplify the error message format," the rewrite touches Gherkin + slice + slice tests + AC + production-code-surface section. Currently the rewrite checklist is implicit — the planner walks the plan section by section. A simple in-plan template ("Affected sections: AC ☐ · production-surface ☐ · slice ☐ · Gherkin ☐ · property tests ☐ · files-touched ☐") would catch the line-37 drift mechanically. Try it on Story 3.3's plan revision step. If it surfaces ≥1 catch on Story 3.3 + 3.4, codify into CLAUDE.md § 6.1 phase 2.
- **Use the recording-fake pattern for service-tier wiring properties more broadly.** The Phase 4 refactor introduced a recording fake (`recordingLedger` captures `asOfDate` and asserts `observedDate === asOf`) to validate that the service forwards arguments verbatim to the port. This pattern catches a whole class of "service silently drops a parameter" bugs that constructed-state fakes can't see. Worth applying proactively in Story 3.3 (Recurring Cost Forecast) which will likely have similar service↔port forwarding contracts.
