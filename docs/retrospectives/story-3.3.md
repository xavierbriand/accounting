Third story of Epic 3. Ships a pure Core forecast service (`RecurringForecastService` + `cadence.ts` helpers) plus a new YAML `recurring:` section with per-rule lifecycle (`validFrom` + optional `validTo` + optional `amendments[]`). No port, no adapter, no ledger reads — variance/validation against actuals is Story 3.4's job. Third epic-3 story to round-trip both `plan-reviewer` (Phase 2) and `code-reviewer` (Phase 4) sub-agents end-to-end. 12 commits on the branch (plan + plan-revision + 10 implementation slices + 1 Phase-4 refactor + this retro). - **Disambiguating contradictory user answers in real time.** User picked "hybrid: configured + ledger validation" for Q1 AND "no ledger reads" for Q4 — incompatible. Caught the contradiction before drafting the plan, asked a focused follow-up that gave three reconciliation options (stage it, bundle it, or push validation entirely to 3.4); user chose option 3. - **Plan-reviewer sub-agent caught a public-API correctness bug pre-implementation.** The initial plan exposed `nextOccurrence(date, cadence)` as a public helper. Plan-reviewer pressure-tested the DoM-clamp behaviour and proved the two-arg form cannot preserve the anchor day across overflow steps (Jan 31 → Feb 29 → Mar 29, not Mar 31). Plan rewritten before Sonnet started: `nextOccurrence` dropped from the public surface; `enumerateOccurrences(validFrom, ...)` ratchets internally from the original anchor. This is a *real* algorithm bug that would have shipped if Sonnet had started from the original plan. The dedicated Phase-2 reviewer surface (codified in story-maint-13) earned its keep on its third epic-3 outing.

- **Plan-reviewer also caught a vacuous property-test formula at plan time.** The original property test #1 used a closed-form count `floor((min - max) / cadenceStep) + 1` — ambiguous over month-vs-day arithmetic and broken under DoM clamp. Plan-reviewer flagged it as flaky-test risk; rewrite split it into per-element well-formedness (#1) + cadence-step regularity (#1b). The rewrite was correct in spirit but the *implementation* of #1b shipped vacuous (see Change below) — the same bug class survived plan-time review and was caught at code-review time. **Lesson:** the plan-revision can specify the right invariant in prose but the implementation can still drift. See Change below.

- **Code-reviewer sub-agent caught two test-honesty issues + the pre-existing CI flake.** Phase 4 retro-check found that property #1b's `expectedMonths % step !== 0` is vacuous when step=1 (any integer % 1 === 0); property #8 claimed the recording-fake pattern from Story 3.2 commit 59639e1 but actually only checked output-containment + inverted-args failure (catches neither argument drop nor argument swap). Both passed green; both `# fails if` claims were unreachable. Phase 4 refactor commit `95ee561` reframed both: #1b now asserts strict delta-equality on consecutive pairs across all three cadences; #8 became a witness-based test (≥32-day window must yield ≥1 occurrence, falsifies on swap/drop). The same R6/R7 honesty class as Story 3.2 — caught and tightened in the same Phase 4 motion. **The reviewer pattern works.**

- **Phase 4 also caught a pre-existing CI flake unrelated to 3.3.** The canonicalize property test had a fast-check assertion `!r.error.includes(content)` where `content` is a 3+ char random string. Random 3-char strings collide with common English words in the error template ("contains" includes "con"). Fast-check found it on Story 3.3's PR. Pre-existing, but blocked merge per DoD §7.1. Fixed under R9 trivial-inline carve-out (≤5 LOC, single file): assert against the FULL tampered string (`withUs`) rather than the bare prefix — meaningful PII-safety witness, no false-positive surface.

Plan-time decision was 10 slices, all enumerated. Sonnet didn't have to make compression judgement calls. (The slice-5 over-implementation deviation below is a different class of issue — see Change.)

- **Retro Try C cross-reference checklist held.** When the plan-reviewer's `nextOccurrence` finding propagated, the rewrite touched: production-code surface (R2 section), property test #4, files-touched table, blind spots, AC-line on amendment vs validTo. All five sections updated in one edit pass. The Story 3.2 retro Try C checklist (AC ↔ Gherkin ↔ slice ↔ production-surface) prevented the "line-37 drift" failure mode 3.2 hit.

## Change

- **Sonnet implemented all three cadences + amendments + validTo in slice 5; slices 6/7/8/9 collapsed to green-on-landing or empty commits.** The plan prescribed monthly-only for slice 5, with amendments/validTo deferred to slice 7 and quarterly/annual to slice 9. Sonnet's `CADENCE_STEP` record + `selectAmount` helper made shipping all three at once "natural"; the deviation note acknowledged this. As a result the planned TDD rhythm (failing test → minimal green ×4) collapsed to one mega-implementation slice + four shells. The slice-5 commit touches 122 LOC across three files — bigger than any planned slice. This is the inverse of retro Try B: Try B says "don't defer compression decisions to the implementer"; here, Sonnet *aggregated* what the plan had explicitly split. **Next time:** when the plan splits incrementality across slices for *deliberate* TDD pacing reasons (not file-size constraints), Sonnet should respect the split even if the helpers are easy to land together. Codify this expectation in the sonnet-implementer agent spec? Not yet — first occurrence; promote if it recurs. Action item A.

- **Property test #1b shipped vacuous despite the plan-time correction.** Plan-reviewer caught the closed-form-count problem at Phase 2; the rewrite specified strict equality `monthsBetween(validFrom, d_i) == i × cadenceStep`. Sonnet's implementation used `expectedMonths % step !== 0` with step hardcoded to 1 — any integer modulo 1 is zero, so the property always passed. The plan said the right thing; the implementation drifted. This is a class of bug the Phase-4 code-reviewer caught (and would catch again), but it shouldn't have shipped. **Lesson:** when a property test's `# fails if` clause cites a specific defect class (here: "two-arg nextOccurrence loses anchor day"), Sonnet should sanity-check that the assertion can FAIL in that defect mode. A cheap mental test: "if I introduced the named bug, would this property fail?" Vacuous assertions never fail. Action item B (codify the sanity check in `.claude/agents/sonnet-implementer.md`).

- **Property test #8 also drifted from the plan's recording-fake spec.** Plan locked the Story-3.2 recording-fake pattern. Sonnet acknowledged in the deviation note: "a stricter implementation would spy on enumerateOccurrences directly to assert the exact values forwarded — deferred." But the test still claimed the recording-fake pattern in its name and `# fails if`. Two defects: (a) the pattern doesn't actually transfer (Story 3.3 has no port; it calls a free-standing function), so the plan was over-specified; (b) the test's name/comments shipped misleadingly. Phase 4 refactor reframed as witness-based and updated the comments. **Lesson:** the recording-fake pattern requires a port for honest interception. For free-standing-function dependencies, witness-based tests (set up an input where any wiring defect produces an observable behavioural difference) are the honest analogue. Update CLAUDE.md or the retro-Try notes to record this nuance? Action item D.

- **Plan over-listed `tests/integration/infra/config/config-service.test.ts` in Files-Touched.** The file was listed with the note "update YAML fixture to include `recurring:` (or empty `recurring: []` to keep tests minimal)." Because `recurring:` is `optional().default([])` in the Zod schema, existing inline YAML fixtures parse cleanly without modification. Sonnet didn't touch the file; tests pass. Plan listed a touch-point that turned out unnecessary. Not a failure mode (better to over-list than under-list), but worth noting: the retro Try C cross-reference checklist would have caught this on the plan side (and could have either kept the touch as defensive coverage or removed it from the list). **Lesson:** when adding a config field as `optional().default()`, the integration-test fixture update is no longer required — note this in the plan template.

## Try

- **Sonnet sanity check on property-test assertion strength.** When Sonnet writes a property test whose `# fails if` clause cites a specific defect class, before commit Sonnet runs the mental "would this assertion fail if I introduced the named defect?" If the answer is "no" (assertion is vacuous, or only checks side-effects, or compares against a re-derivation of the same bug), tighten the assertion before commit. Try this on Story 3.4's plan; if it surfaces ≥1 catch, codify in the sonnet-implementer agent spec. Action item B.

- **Honest test-pattern selection: recording-fake vs witness-based.** When a plan calls for a wiring-contract property test, Sonnet (or Plan-agent) chooses between two patterns based on architecture: **recording-fake** if the dependency is a port (intercept call args); **witness-based** if the dependency is a free-standing function (engineer an input whose output diverges under any wiring defect). Story 3.3 should have used witness-based from the start. Try this disambiguation explicitly in Story 3.4's plan; if it sticks, fold into CLAUDE.md § 5 alongside the existing test-tier table. Action item D.

- **Plan-time AskUserQuestion for "load-bearing decisions" graduates from Try to Keep.** Three stories in (3.1, 3.2, 3.3), the pattern has caught real coherence bugs every time. Rather than re-trying it on every story, codify in CLAUDE.md § 6.1 phase 1 as a numbered step: "1.5: Identify and lock the 2–4 load-bearing decisions via AskUserQuestion before drafting the plan." Action item C.
