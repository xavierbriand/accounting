## Change

- **Sonnet hit context-limit mid-slice-7 and left an uncommitted interface change.** Story 3.1's retro Try ("re-spawn Sonnet on context-limit") covers the recovery path, and the re-spawn went cleanly. But the post-recovery state had a subtle bug: the re-spawned Sonnet ran `npm run build` locally and saw green (because the working tree was complete), then committed only the production-code references to `state.targetDate` — not the interface declaration adding `targetDate` to `BufferState`. Locally everything looked fine; on origin, CI was RED. **Lesson**: when re-spawning Sonnet to resume mid-slice, the resume brief MUST explicitly say "git status before commit; ensure ALL uncommitted modifications are part of the slice or restored." The brief I wrote did say "Fix the BufferState literal in `buffer-state-service.ts` to include `targetDate: bucket.targetDate` so the build passes" — but that referenced the *type assertion*, not the interface declaration. The re-spawned Sonnet correctly did the type assertion (in `buffer-state-service.ts`) and assumed the interface was already in HEAD. Action item A: tighten the resume brief template in `.claude/agents/sonnet-implementer.md` to require an explicit `git diff HEAD --name-only && git status -uno` snapshot before the first commit on resume.

- **Property #1 had a hidden floating-point trap.** When I rewrote Property #1 from "Math.floor + ±1" to "Money.allocate exact", I introduced a new bug: the property recomputed `1 - ratio2` in the test body, which differs from `1 - ratio` in IEEE-754 (e.g., `1 - 0.9 = 0.09999...998`, not 0.1). Production reads ratios from the splits config (computed once at fixture creation); the test recomputed them differently. Fast-check found the discrepancy on the first non-symmetric ratio. Fix: compute ratios ONCE in the generator and reuse for both window construction and assertion. **Lesson**: when writing a property whose mechanism depends on floating-point ratios, the ratio values must be byte-identical between the fixture-builder side and the assertion side. Recomputation via algebraic identity (`1 - r`) is a footgun. Logged in the Phase-4-refactor commit body for future readers.

- **Misleading "first higher-weight element" comment for the LRM tie-breaker.** Sonnet's slice-5 comment for the 60/40 unit test claimed dinero.js gives the remainder cent to "the first higher-weight element" — that's not standard LRM (which awards by largest fractional part) and not what dinero.js actually does (it awards by input order). The test's expected values (780/519) were correct for dinero.js v2 behaviour; the comment misattributed the cause. Code-reviewer caught it; rewrote in Phase 4. **Lesson**: when documenting library-specific behaviour, name the library and version explicitly, not the general algorithm.
