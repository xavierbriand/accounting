## Change

- **A. The audit-completeness gap reproduced for the second consecutive R15 story — this crosses the codify threshold.** [Story-maint-21 retro item C](story-maint-21.md) flagged: *"when a changelog audit table claims completeness, cross-check the bullet count against the real upstream changelog before asserting it."* This story's plan made the identical mistake at a larger scale: its audit table claimed "All other `@cucumber/*` changelog entries... scoped to PHP/.NET/C++/Java, not the JS/TS package we consume" — factually wrong for one entry (`@cucumber/gherkin` 37.0.0's unscoped "Require messages v31 or greater" breaking change, which nests a nested `@cucumber/messages@33.0.4` copy distinct from our top-level `27.2.0`). The `code-reviewer` Phase 4 agent caught it by independently `curl`-ing the real upstream `CHANGELOG.md` rather than trusting the plan's summary — exactly the check both this story's own Phase 4 prompt and story-maint-21's retro asked for, and exactly the check the *plan-drafting* step (this session, before Phase 4) still skipped. **Empirically harmless** (both `@cucumber/messages` copies' `IdGenerator.d.ts` are byte-identical; `test:harness` 252/252 green both times) — but "harmless in this instance" isn't the same as "the audit process is sound." Two identical near-misses in two consecutive stories is a pattern, not a coincidence: **a real changelog audit needs to be produced by diffing the actual `CHANGELOG.md` file across the full version range (e.g. `curl` + `grep -c BREAKING`), not by reading a Dependabot PR body's excerpted release notes or summarizing from memory.** Neither this session's own pre-plan research (delegated to an agent, but the agent's own report was accepted without an independent completeness cross-check) nor the plan-drafting step caught it — only Phase 4 did, on both occasions.
- **B. Two more soft findings reproduced identically from story-maint-21** — the R23 story-id-uniqueness narration gap, and the R15/§6-lane-table reconciliation gap. Both fixed in this story's plan text (the narration gap) or filed as a follow-up issue ([#200](https://github.com/xavierbriand/accounting/issues/200), the lane-table gap — two identical data points now, past this repo's own "codify once it reproduces" bar). **These are cheap, mechanical gaps that a plan-writing checklist item would catch before Phase 4 ever runs** — worth turning "check R23 explicitly" and "the plan's lane framing must acknowledge R15" into an actual line in [docs/templates/maintenance-sub-loop.md](../templates/maintenance-sub-loop.md) rather than relying on Phase 4 review to keep re-discovering them.
- **C. A git operational surprise: rebasing a branch whose earlier PR was squash-merged (and whose remote ref was then auto-deleted) needs `rebase --onto`, not a plain `rebase`.** This session reused the same long-lived branch (`claude/dependabot-pr-maintenance-53c58f`) across both story-maint-21 and story-maint-22. After PR #197 (story-maint-21) was squash-merged, `git rebase origin/main` tried to replay the *original, unsquashed* maint-21 commits on top of `origin/main` (which now contained their squashed equivalent as one new commit) — producing an add/add conflict on `docs/plans/story-maint-21.md` that looked like a real content conflict but was actually "these commits are already merged, don't replay them." Resolved with `git rebase --onto origin/main <old-branch-tip> HEAD` to skip straight to the new commits. Then the plain `git push` also failed (`stale info`) because the remote branch had been auto-deleted by GitHub's `deleteBranchOnMerge` setting after the squash-merge, and a `--force-with-lease` push against a ref that no longer exists on the remote fails differently than one might expect — a plain `git push origin HEAD` (no force needed) recreated the branch cleanly. **Not currently written down anywhere in [CLAUDE.md § 6.4.1](../../CLAUDE.md)'s push protocol**, which assumes a branch's own PR is still open when the next rebase happens. Worth a one-line addition: when reusing a branch name across sequential stories in one session, expect the previous story's PR to have merged-and-deleted the remote ref, and use `rebase --onto <old-tip> origin/main HEAD` + a plain `push` (not `--force-with-lease`) to recover.

## Try

- **Add an explicit "diff the real upstream CHANGELOG.md, don't trust the PR body excerpt or a summarized research report" step to the maintenance-sub-loop checklist or a plan-writing habit, specifically for major-version breaking-change audits.** This is now 2/2 R15 stories where the *drafting* pass missed a real changelog entry that Phase 4 review caught by going to the primary source. Concrete mechanism to try next time: `curl` the changelog and `grep -c BREAKING` (or equivalent) across the full bumped range as a literal command in the plan's probe section, not a prose claim of having "checked."
- **Observation target: does the R23-narration and R15/lane-table gaps stop recurring now that #200 is filed and this story's plan fixed both inline?** If a *third* R15 story reproduces either, that's a stronger signal the fix needs to land in [docs/templates/maintenance-sub-loop.md](../templates/maintenance-sub-loop.md) directly rather than being re-derived per-story.
- **If this session (or a future one) reuses a long-lived branch across another sequential maintenance story, confirm the `rebase --onto` + plain-`push` recovery pattern from Change C works the same way a second time** — one data point so far.
