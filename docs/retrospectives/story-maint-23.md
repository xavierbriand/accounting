# Story maint-23 retrospective

**PR:** [#205](https://github.com/xavierbriand/accounting/pull/205)  **Closed:** pending merge

Adds a homegrown local ESLint rule set (`eslint-rules/test-smells/`, 9 rules) grounded in ["Investigating Test Smells in JavaScript Test Code"](https://doi.org/10.1145/3482909.3482915) (Jorge, Machado, Andrade, SAST'21), plus two `code-reviewer.md` checklist bullets for the smells lint can't automate. Reduced lane — dev-tooling + agent-spec, no `src/core` touched.

## Keep

- **Implementing each rule against the real suite immediately, not batched, caught 4 genuine false positives during development instead of after merge.** Every rule was wired at `warn`, swept with `npm run lint` against the actual 800+-test suite the moment it landed, and inspected. That surfaced: `no-redundant-assertion` flagging a legitimate hash-determinism check (`expect(hash(x)).toBe(hash(x))`), `duplicate-assert` flagging a migration-idempotency re-check with a state-changing call in between, and `no-unasserted-test` flagging 49 fast-check property tests whose own `fc.assert(fc.property(...))` idiom *is* the assertion (predicate return value, no inner `expect()`). All four fixed in-PR with a regression-locking `RuleTester` fixture added for each. Worth keeping as standard practice for any future "write a new lint rule" story: sweep the real corpus mid-development, not just at the end.
- **The paper's empirical evidence (frequency + quality-correlation data) was directly actionable for scoping, not just narrative color.** Dropping Magic Number Test outright (domain's correct Money representation is bare integer-cent literals — the paper's own most-correlated smell was the wrong fit for this specific codebase) and keeping Assertion Roulette lenient/`warn`-forever (the paper explicitly notes it's rare in Chai/Jest-style `expect()` code, which Vitest shares) were both plan-time calls the real-suite sweep later confirmed: 0 Magic-Number-shaped false-positive risk avoided, and 41 real Assertion-Roulette hits that would have been noisy at `error`.
- **Phase 2 `sibling-overlap` caught a real story-id collision before any code review round-trip.** The plan was originally `story-maint-22`; a worktree-local `ls docs/plans | grep maint-22` looked clear (the branch was one commit behind `origin/main`), but `sibling-overlap` traced it to an already-merged, unrelated PR (#199) via `git fetch origin main` + a tree check — exactly the class of gap R23 exists to catch, and exactly the kind of check a worktree-local grep structurally cannot perform. Renumbered to `story-maint-23` before any implementation work, at near-zero cost. Same "agent isn't rubber-stamping the plan" value story-maint-21's retro noted for the same agent.

## Change

- **A. Plan-time "confirmed-zero baseline" claims (via manual, narrowly-scoped greps) were less reliable than the actual `RuleTester`-and-real-sweep loop.** The Phase-1 exploration's false-positive hunting (grepping for `expect(true).toBe(true)`-style tautologies, `.skip`/`.todo` usage, `better-sqlite3` imports) was thorough enough to get 4 of the 9 rules right at `error` from the first commit, but missed 3 more subtle patterns (determinism re-checks, idempotency re-checks, the `fc.assert` idiom) that only surfaced once the rule was actually implemented and run against real code. Not a planning failure exactly — the plan's own rollout design (start most rules at `warn`, promote after a sweep) anticipated this — but worth naming precisely: a manual grep during planning is a *prior*, not a substitute for actually running the mechanism against the corpus.
- **B. The `CLAUDE.md § 8` mid-story transient (rule tag cited in a `.claude/` spec before its table row exists, then a table row before its retro exists) reproduced twice in this story, confirming it's a real, recurring pattern rather than a one-off.** `drift-scan --all` flagged `claude-stale-tag: R29` after the `code-reviewer.md` commit (R29 cited, no table row yet), then `table-only: R29` after the `CLAUDE.md` § 8 row was added but before this retro file existed. Both resolved by the time this PR is complete, per the "same PR" (not "same commit") reading of the § 8 preamble — but this is the second story to hit this exact category of gap (see [story-maint-21](story-maint-21.md) § Change A for the sibling case: a *hard* Gherkin-gate false-positive from the same underlying tension between "one story landing in staged commits" and "checks that assume the final state"). [Issue #178](https://github.com/xavierbriand/accounting/issues/178) already tracks the general "mid-story table-only transient" direction question — this story is a second concrete data point for it, not a new issue.
- **C. `conditional-test-logic` produced far more real hits (151) than any other rule, and includes at least 2 legitimate-but-unexcluded categories mixed in with likely-genuine smells.** Deliberately left un-narrowed in this story (see [issue #206](https://github.com/xavierbriand/accounting/issues/206)) rather than self-authoring exceptions without a human triage pass — the other 3 false-positive fixes in this story were all *unambiguous* (the flagged code was clearly not a smell by any reasonable reading); `finally`-cleanup guards and `fc.property` preconditions are genuinely debatable, and the rule stays advisory (`warn`) specifically so that debate doesn't have to be resolved under this story's own time pressure.
