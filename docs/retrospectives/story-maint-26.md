# Story maint-26 retrospective

**PR:** [#224](https://github.com/xavierbriand/accounting/pull/224)  **Closed:** pending merge  **Closes:** no tracked issue (surfaced directly from a user bug report in chat)

Closes a completeness gap in FR20/story-4.4b: Commander's own parse-time failures (missing required option/argument, unknown option, excess arguments) were caught by Commander's parser before any command's action handler ran, entirely bypassing the `--json` failure envelope — plain prose to stderr, exit 1, unconditionally. Root cause of the original bug report (`npm run ingest --json` with `-f` omitted) was actually two layered issues: an npm flag-forwarding footgun (user-educated in chat, no code fix) plus this genuine gap, which the story-4.4b `correct.feature` scenario comment had already flagged as a known, deliberate omission. Fix: `program.exitOverride()` + a catch around `program.parse()` in `src/cli/program.ts`, translating the known bad-usage `CommanderError` codes into the existing `INVALID_ARGUMENT` envelope, scoped to the 5 `--json`-capable commands, exit 2 unconditionally. Diff: 5 files, ~475 insertions (mostly the new subprocess test file and the plan), 10 commits.

## Keep

- **The Phase-2 `sibling-overlap` risk note ("re-verify against commander 15 when #223 lands") paid off exactly as written.** [PR #227](https://github.com/xavierbriand/accounting/pull/227) merged mid-implementation; the plan's own Risks table meant nobody was surprised, and the implementer empirically re-checked every `CommanderError` code/message this story depends on against commander 15.0.0 (byte-identical to 14.x) as a matter of course rather than as a fire-drill. Flagging a cross-story dependency risk *before* it materializes, with a named "whichever merges second re-verifies" protocol, is worth repeating.
- **A subagent correctly refusing to proceed on a text-message "approval" was the harness's safety boundary working exactly as designed, not a bug to route around.** Mid-Phase-3, `sonnet-implementer` detected that plan mode was still session-wide active (a prior `ExitPlanMode` call had errored at the transport layer without actually clearing the flag) and refused to write code despite an explicit "reviewed and approved, proceed" message from the coordinating agent — citing, correctly, that only the permission system (not another agent's text) can lift plan mode. That refusal was trusted rather than argued with; the actual fix was to re-run `ExitPlanMode` properly and get the real harness confirmation before resuming. See § Change/new rule below.
- **Empirical verification over documentation-reading, twice.** Both at Phase 1 (reading `node_modules/commander/lib/command.js`'s `copyInheritedSettings`/`_exit` directly rather than trusting commander's own docs on `exitOverride()`/`configureOutput()` inheritance) and again post-#227 (re-running the same kind of direct check against commander 15's actual installed source, not its changelog), the session reached for the real source over the library's prose. This caught a real ordering bug (see § Change) that reading commander's public docs alone likely would not have.
- **The Phase-4 fix-now round caught a genuine scope-leak bug before merge, not after.** `code-reviewer` found that the exit-2 branch fired for *any* command hitting the allowlisted Commander codes, not just the 5 `--json`-capable ones — silently changing `migrate`'s malformed-invocation exit code from 1 to 2, untested and outside the story's actual intent. Fixed via normal TDD (failing `migrate --nope` regression test → green), not a quick unreviewed patch.

## Change

- **New rule (R32): after `ExitPlanMode` errors at the transport layer, verify the harness's actual exit-confirmation before treating plan mode as cleared.** The stuck flag didn't block the main loop's own subsequent tool calls (`Write`/`Edit`/`Bash` for git operations and `gh pr create` all proceeded normally) — it only surfaced when a spawned subagent needing write tools checked and correctly refused. That's a dangerous asymmetry: a less careful main-loop agent could have kept dispatching subagents indefinitely, each one either (a) correctly refusing, silently stalling the story, or (b) — worse — being argued past the restriction by a coordinator insisting its own text approval was sufficient. Codified in CLAUDE.md § 6.1 (see this PR's CLAUDE.md diff) and § 8.
- **The plan's own code sketch didn't translate a correct research finding into an unambiguous instruction.** Phase 1 planning *did* read `copyInheritedSettings()` and correctly noted it "propagates to all subcommands... confirmed by reading node_modules/commander/lib/command.js" — but the plan's prose described calling `exitOverride()` "once" alongside "wrap the final `program.parse()`... in a try/catch," in an order that reads as "both near the bottom of the file." The implementer had to independently rediscover (via direct empirical testing, not just re-reading the same source file) that `exitOverride()` must run *before* any `.command()` registration, or the copied `_exitCallback` is `null` and the whole fix silently no-ops. The research was right; the plan's translation of that research into a concrete code-placement instruction was ambiguous. Second occurrence in this repo's visible retro history (per story-maint-23's retro) of "a plan's prose hedges on something a two-minute check would have nailed down" — worth watching for a third before treating it as a pattern needing its own rule.

## Try

- **When a plan cites a specific mechanism finding (e.g., "X must happen before Y, per reading the library's source"), state the resulting placement/ordering constraint as an explicit instruction in the plan's code sketch, not just as a citation of the research.** Cheap to do at planning time; expensive to rediscover during implementation.
- **The `/loop` dynamic-mode + `ScheduleWakeup` self-pacing pattern for "wait on an external, human-gated PR merge, then act" worked cleanly** (checked PR #227's merge state every ~20 min, no fixed interval needed, cancelled cleanly once the dependent work completed the wait organically anyway). Worth reaching for again for future cross-story dependencies rather than either blocking synchronously or relying on the user to remember to ping back — though this run made the wait itself moot (the implementer's own fix-now round happened to straddle the #227 merge and handled it inline), so the pattern is validated by design, not yet by a case where it was the thing that actually closed the loop.
