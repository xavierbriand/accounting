## Keep

- **Probing beats changelog-reading, twice more, on the two riskiest packages in this bump.** Both exclusions were confirmed by actually installing the package and running the real toolchain, not by reading prose: `typescript@7.0.2` installs and builds cleanly but crashes `npm run lint` outright (`typescript-estree` reaching into changed TS7 compiler internals) — a failure a changelog summary would never have surfaced, and confirmed as a *current*, not transient, incompatibility by checking that `typescript-eslint@8.64.0` (`latest`, the exact version this same bump targets) still pins `typescript: ">=4.8.4 <6.1.0"` upstream today. `pixelmatch@7.2.0`'s conflict with `quickpickle`'s peer range was confirmed the same way (`npm view`), not assumed from precedent. Same discipline story-maint-21/22 established, now 4/4.
- **Delegating the id-collision detection to the Phase 2 `sibling-overlap` agent caught something a purely mechanical branch-name grep couldn't.** The maintenance sub-loop's R23 check (`git ls-tree` + `gh pr list --json headRefName`) came back clean at draft time because it only inspects branch *names*; [PR #224](https://github.com/xavierbriand/accounting/pull/224) — an entirely unrelated concurrent story with its own `docs/plans/story-maint-26.md` — had a branch name (`claude/cli-error-output-format-cecd23`) that gives no hint of its claimed story id at all. Only an agent that actually reads live plan-file *content* across open PRs found the collision. This is a genuine gap in R23's mechanical form, not a one-off miss (see § Change).
- **Fixing a real audit-completeness gap once a review agent surfaced it, rather than arguing it away.** Phase 4 review found the plan's Conclusion line ("no in-repo import of any of them outside their own tooling role") was independently falsifiable — `fast-check` is imported in ~25 test files, `eslint`/`typescript-eslint` in `tests/_helpers/eslint-rule-tester.ts`. The bump's safety conclusion still held (these are pre-existing test-tooling import sites, not new surface, confirmed by the unchanged test suite), but the plan's *stated premise* was wrong. Rewrote it to name the real import sites accurately rather than downplay or dispute the finding.

## Change

- **A. Story-id renumbering mid-flight required a full commit-history rewrite, not a forward-fix commit — and the first attempt (a forward-fix rename commit) was insufficient.** When the collision with PR #224 was found, the initial fix was a new commit renaming `docs/plans/story-maint-26.md` → `story-maint-28.md` and explaining the rename in its body — leaving the three earlier commits' *subjects* still saying `(story-maint-26)`. This passed local review but failed CI's `dod-check` hard gate (`missing-story-id` on all three older commits, since `dod-check` matches the *current* story id — derived from the plan file the PR actually adds — against every commit subject in the PR, with no tolerance for a superseded id even with an explanatory rename commit on top). Fix: `git reset --soft` back to the merge-base and recommitted all three with the corrected id from the start, then `git push --force-with-lease` (safe here — solo-authored, unmerged, still-draft branch, no collaborators). **This needed doing twice** (see Change B) — the first rewrite fixed the id but didn't yet include the Phase 4 findings' text corrections, which required a second reset-and-recommit pass once the code-reviewer's findings came back.
- **B. Phase 4 review findings on a zero-code-change story still required a real commit-history rewrite, exposing that "fix-now" doesn't automatically mean "add a fixup commit" for this story shape.** Two separate rounds of Phase 4 findings (factual audit-completeness error; R30 commit-subject wording gap on the sibling story) each required rewriting already-pushed history rather than layering a new commit on top, because the story's own commit *messages* (not just its code) were the thing under review, and R15's collapsed 3-commit shape has no slot for a "fix the previous commit's wording" commit without breaking `countSlices`'s exemption logic. Two `git reset --soft` + recommit + `--force-with-lease` cycles this session (one for the rename, one for the Phase 4 text/justification fixes) — both against a solo, unmerged, draft branch, so low-risk, but worth naming as a recurring shape for this story type: **when Phase 4 findings touch commit *subjects or bodies themselves* (not just plan prose), expect a rewrite, not a fixup commit.**
- **C. R30's exact-match `PREP_COMMIT_SUBJECT` regex (`plan + P1/P2/P3 review`) silently rejects a paraphrase, and this session's own first-draft commits used a different paraphrase ("plan + Phase 2 review") without noticing.** `harness/dod-check/lib/commit-subject.ts:70`'s regex requires the literal substring `plan + P1/P2/P3 review` — not just "some review happened" language. This session's original prep commits on *both* sibling stories (story-maint-27 and this one) used "plan + Phase 2 review" instead, which doesn't match, inflating the reported slice count. Phase 4 review caught it here and cross-checked: **every Reduced-lane maintenance-story prep commit since R30 landed (story-maint-25, story-maint-27, and this one before the fix) has the identical wording drift.** This is a distinct, more mechanical angle than [issue #200](https://github.com/xavierbriand/accounting/issues/200) (which is about R15 not being a recognized envelope token at all) — R30's exact-phrase requirement is satisfiable today, sessions are just drifting off the literal text. Not re-filed as a new issue since the fix is trivial once known (copy the literal phrase); flagged here so the next maintenance story's author checks R30's text directly rather than paraphrasing from memory.

## Try

- **Extend the maintenance-sub-loop R23 check to grep in-flight open PRs' *added plan-file paths* (e.g. `gh pr diff <n> --name-only | grep docs/plans/story-`), not just branch names.** This session's collision would have been caught before drafting, not after Phase 2, if the check had looked at what files each open PR actually adds rather than trusting branch-name conventions. Concrete mechanism: for each open PR, `gh pr diff <n> --name-only` and check for any `docs/plans/story-<id>.md` path matching the id about to be claimed.
- **Watch whether "plan + P1/P2/P3 review" wording drift (Change C) recurs on the next maintenance story now that this retro names it explicitly.** If it does, the fix is worth promoting from "watch for it" to an explicit line in [docs/templates/maintenance-sub-loop.md](../templates/maintenance-sub-loop.md) quoting R30's exact required phrase.
- **If a third R15-shaped story needs a commit-history rewrite mid-review (Change B), consider whether R15's collapsed shape needs an explicit "fixup slot" convention** (e.g. permitting one additional `fixup!`-style commit that's `countSlices`-exempt) rather than requiring a full history rewrite every time Phase 4 review finds a wording issue in an already-pushed commit.
