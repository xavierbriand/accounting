Story maint-17 — `IdempotencyService.filterNew()` in-batch sequence tie-breaker shipped. Bug fix for a `transactions.idempotency_hash` UNIQUE-constraint failure that rolled back the entire `saveBatch` when a real BPCE export contained two same-day rows with identical (sourceAccount, direction, amount, payee). Cause: BPCE's day-only `Date operation` makes the canonical-key inputs collide for legitimate repeat transactions on the same day; `filterNew` only deduplicated against the DB, not within the input batch. Fix: 1st occurrence of a canonical hashes unchanged (backward-compatible with every already-committed row); 2nd+ occurrences hash `canonical + <US> + "#" + seq` before SHA-256. No schema or migration change.

Implementation: 6 commits (plan + 4 code + retro). Plan-reviewer Phase-2 surfaced 8 findings (6 adopt, 1 acknowledge, 1 mistakenly-rejected — corrected in Phase-4). Code-reviewer Phase-4 surfaced 5 substantive findings; refactor commit fixed all in-PR (US-constant deduplication, R6 wording on test 2b, property-test vacuous-truth return, redundant-scenario consolidation, suggestion-log integrity correction). One follow-up filed: [#117](https://github.com/xavierbriand/accounting/issues/117) (test-infra: extract `writeStubYaml` extras + scope-narrow `Given a fresh migrated DB`).

Two retro lessons worth lifting: (1) privacy-rule scope broadened from "merchant strings only" to "all transaction-private data" — vendors, accounts, sums, calendar dates, filenames embedding account ids — pending promotion to CLAUDE.md § 8 R20; (2) Phase-2 reviewers should ground "no precedent" rejections in a falsifiable evidence pointer (grep result, line numbers) rather than a bare assertion.
