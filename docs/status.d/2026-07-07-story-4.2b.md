Story 4.2 (Correction) fully shipped — 4.2b lands the `correct` CLI command, completing the split from 4.2a. Explicit-flag command (`--amount`, `--category`, `--date`, `--description`, `--reason` required, `--json`): loads the original transaction, calls 4.2a's `CorrectionService.correct`, persists via `saveCorrection`, and records `TransactionCorrected` at the app boundary (B1, mirroring ingest's `commitBatch`). `--amount` parses string→integer-cents directly (no float intermediate, per security-checklist.md); `--date` splices a bare `YYYY-MM-DD` onto the original's time-of-day + UTC offset (no `Date` round-trip, DST-safe). `CorrectionService` gained two new guards — reject correcting a `reversal`-kind transaction (new invariant 9, reconciled into the [story-4.0 model note](../domain/model-notes/story-4.0.md)) and require ≥1 changed field — plus closed #185 (absent-vs-empty-string field detection in `CorrectionChanges`). `changedFields` output relabels `"account"` as `"category"` for the CLI user (the recorded `TransactionCorrected` event keeps Core's `"account"` vocabulary). `ddd-modeler` Mode B: 0 conformance violations. Audit-event atomicity stays B1 (#180 remains the dedicated UnitOfWork story).
